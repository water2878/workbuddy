## 修复图片消息导致上下文压缩（根本原因）

**问题发现**：
- 客户发送图片时，上下文必定被压缩
- 日志显示图片消息被发送到WorkBuddy时携带了图片数据（base64）
- `send_text_and_image_to_workbuddy()` 通过剪贴板粘贴图片到WorkBuddy
- WorkBuddy将图片base64数据包含在上下文中，导致token爆炸

**根本原因**：
- bridge把图片数据直接粘贴到WorkBuddy输入框
- 图片base64编码后可达数MB，远超模型上下文限制
- 即使做了文件系统整理，单张图片数据就能撑爆上下文

**修复内容**（`weflow_wb_bridge.py`）：
1. **图片消息处理**：不再调用 `send_text_and_image_to_workbuddy()`，改为只发送图片路径文本
2. **消息格式**：`[昵称]: [图片]\nPath: xxx.jpg\nURL: xxx`
3. **Fallback处理**：文本消息内容含`[图片]`时也获取图片路径发送路径文本
4. **AI识别流程**：AI收到路径后 → 提取Path → 调用 `/api/image/match` 分析图片

**修改前后对比**：
- 修改前：发送图片base64数据（数MB）→ 上下文压缩
- 修改后：发送图片路径文本（几十字节）→ 上下文正常

**验证**：
- `python -m py_compile weflow_wb_bridge.py` 语法检查通过

## 修正合同生成流程（重要）

**问题发现**：
- 客户说"来一个合同"时，边收集信息边生成合同
- 生成的合同内容不完整：缺地址、缺面板级别、公司名称不对
- 面板产品显示错误：型号"面板E0"、概述"智能升降桌"

**正确流程**：
1. **客户说"要合同"** → 先问"需要生成合同吗？"
2. **收集完整信息**（必须全部确认后再生成）：
   - 公司全称（不是微信昵称）
   - 收货地址
   - 联系人 + 电话
   - 产品明细（型号、数量、颜色、价格）
   - 面板规格（尺寸、厚度、颜色、E0/E1级别、价格）
   - 交期
3. **信息确认无误** → 生成合同
4. **只发文字通知**："合同已生成，合同号XXX，已提交审批"

**错误做法（禁止）**：
- ❌ 边收集边生成（导致信息不全）
- ❌ 用微信昵称代替公司名
- ❌ 面板不写级别和价格
- ❌ 不写收货地址

**修复内容**：
1. **更新合同生成器**（`contract/contract_generator.py`）：
   - 面板产品型号直接显示"E0"/"E1"（不是"面板E0"）
   - 面板产品概述显示"面板"（不是"智能升降桌"）
2. **更新 MEMORY.md 合同流程章节**
